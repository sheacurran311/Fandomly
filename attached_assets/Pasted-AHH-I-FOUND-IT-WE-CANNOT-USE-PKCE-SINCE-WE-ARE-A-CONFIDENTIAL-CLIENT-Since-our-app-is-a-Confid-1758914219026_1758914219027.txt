AHH! I FOUND IT! WE CANNOT USE PKCE SINCE WE ARE A CONFIDENTIAL CLIENT!

Since our app is a Confidential Client in the X developer portal, the correct path is the standard OAuth 2.0 Authorization Code grant without PKCE, using our client secret. Below is a precise, copy-pasteable setup for both the code exchange and refresh, plus the gotchas that typically trigger unauthorized_client.

1) Authorization step (browser redirect):
You should not send a code_challenge/code_verifier when you’re a confidential client.

GET https://twitter.com/i/oauth2/authorize
  ?response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=YOUR_REDIRECT_URI
  &scope=tweet.read%20tweet.write%20users.read%20offline.access
  &state=RANDOM_CSRF_STRING

2) Exchange authorization code → tokens (server, with secret)

You have two valid ways to authenticate the client. Pick ONE and use it consistently for both exchange and refresh.

Option A — Basic Auth (recommended for simplicity):
Header: Authorization: Basic base64(client_id:client_secret)
curl -X POST 'https://api.x.com/2/oauth2/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -H 'Authorization: Basic BASE64_CLIENTID_COLON_SECRET' \
  --data-urlencode 'grant_type=authorization_code' \
  --data-urlencode 'code=AUTH_CODE_FROM_CALLBACK' \
  --data-urlencode 'redirect_uri=YOUR_REDIRECT_URI'

Option B — Client credentials in body (no Basic header):
curl -X POST 'https://api.x.com/2/oauth2/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'grant_type=authorization_code' \
  --data-urlencode 'code=AUTH_CODE_FROM_CALLBACK' \
  --data-urlencode 'redirect_uri=YOUR_REDIRECT_URI' \
  --data-urlencode 'client_id=YOUR_CLIENT_ID' \
  --data-urlencode 'client_secret=YOUR_CLIENT_SECRET'

DO NOT include code_verifier here; that’s PKCE-only.

3) Refresh tokens (server, with secret)

Remember: refresh tokens rotate. Every successful refresh returns a new refresh_token; you must store it and discard the old one.

Option A — Basic Auth (match your choice above):
curl -X POST 'https://api.x.com/2/oauth2/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -H 'Authorization: Basic BASE64_CLIENTID_COLON_SECRET' \
  --data-urlencode 'grant_type=refresh_token' \
  --data-urlencode 'refresh_token=THE_CURRENT_REFRESH_TOKEN'

Option B — Client credentials in body:
curl -X POST 'https://api.x.com/2/oauth2/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'grant_type=refresh_token' \
  --data-urlencode 'refresh_token=THE_CURRENT_REFRESH_TOKEN' \
  --data-urlencode 'client_id=YOUR_CLIENT_ID' \
  --data-urlencode 'client_secret=YOUR_CLIENT_SECRET'

DO NOT include scope or code_verifier on refresh.

4) Minimal Node/Express handlers (Confidential client)
Exchange (authorization_code → tokens):
import fetch from "node-fetch";
import { Buffer } from "node:buffer";

function basicAuth(clientId: string, clientSecret: string) {
  return "Basic " + Buffer.from(`${clientId}:${clientSecret}`).toString("base64");
}

export async function exchangeCode(req, res) {
  const { code, redirect_uri } = req.body; // from your callback
  const auth = basicAuth(process.env.X_CLIENT_ID!, process.env.X_CLIENT_SECRET!);

  const body = new URLSearchParams({
    grant_type: "authorization_code",
    code,
    redirect_uri,
  });

  const r = await fetch("https://api.x.com/2/oauth2/token", {
    method: "POST",
    headers: {
      "Content-Type": "application/x-www-form-urlencoded",
      "Authorization": auth,
    },
    body,
  });

  const text = await r.text();
  if (!r.ok) {
    console.error("X token exchange failed", { status: r.status, body: text });
    return res.status(r.status).json({ error: "exchange_failed", detail: text });
  }

  const json = JSON.parse(text);
  // Persist json.access_token, json.refresh_token, json.expires_in, json.scope, etc.
  return res.json(json);
}

REFRESH:
export async function refreshTokens(req, res) {
  const { refresh_token } = req.body; // your stored (latest) refresh token
  const auth = basicAuth(process.env.X_CLIENT_ID!, process.env.X_CLIENT_SECRET!);

  const body = new URLSearchParams({
    grant_type: "refresh_token",
    refresh_token,
  });

  const r = await fetch("https://api.x.com/2/oauth2/token", {
    method: "POST",
    headers: {
      "Content-Type": "application/x-www-form-urlencoded",
      "Authorization": auth,
    },
    body,
  });

  const text = await r.text();
  if (!r.ok) {
    console.error("X refresh failed", { status: r.status, body: text });
    return res.status(r.status).json({ error: "refresh_failed", detail: text });
  }

  const json = JSON.parse(text);
  // IMPORTANT: rotate stored refresh_token to json.refresh_token atomically
  return res.json(json);
}

5) Common pitfalls that cause unauthorized_client or 401:
Mixed flows: If your authorize request omitted PKCE (correct for confidential), but your token request includes code_verifier, the server can reject it. Remove any PKCE fields entirely.

Missing client authentication: For confidential flows you must authenticate the client on the token endpoint (either Basic header or client_id+client_secret in the body).

Redirect URI mismatch: The redirect_uri in the token request must match the one used in the authorize request and one of the URIs configured in the portal.

Wrong app permissions: In the X developer portal, ensure User authentication settings are enabled and the app permissions align with your scopes (Read/Write if you need tweet.write).

Scope changes midstream: Don’t try to change scopes at refresh; request your full set (incl. offline.access) on the initial authorize.

Token rotation not handled: Reusing an old refresh token → invalid_grant. Always store the new one from the refresh response.

6) Operational recommendations
Atomic storage: When you refresh, write the new access_token + refresh_token in a single transaction to avoid race conditions.

Early refresh: Compute expires_at = now + (expires_in - 60)s and refresh a minute early to ride out clock skew.
